# cli-skip-questions Specification

## Purpose

`cli-skip-questions` capability は、mspec ワークフローにおける「ステップの明示的スキップ」(`mspec skip`) と「フェーズごとの質問テンプレ取得」(`mspec questions`) の責務を担う。`mspec skip` は typo/docs 等の軽微変更で不要なステップを安全に飛ばすため、`workflow.yaml` の `skippable: true` ステップに限定して理由付きで省略を許可し、後続ステップが `requires` を解決できるようプレースホルダ MD を生成、`readme.md` と `skip-log.json` に追跡情報を残し、Constitution Check の検証も免除する。`mspec questions` は CLI 同梱の bank とプロジェクト固有 bank を 2 階層でマージし、9 カテゴリ・`when` 条件・`dynamic` options を踏まえた質問セットを LLM スキルへ提供する。

## Requirements

### Requirement: FR-001 — Allow skip only for `skippable: true` steps

The system MUST refuse `mspec skip <step-id>` and return a non-zero exit code unless the target step is declared with `skippable: true` in `workflow.yaml`.

#### Scenario: Skipping a non-skippable step fails
- GIVEN `workflow.yaml` で step `proposal` が `skippable: false` と宣言されている
- WHEN ユーザーが `mspec skip proposal --reason "typo only change"` を実行する
- THEN コマンドは非ゼロの終了コードで中断する
- AND `readme.md` には `## Skipped Steps` の追記が行われない

#### Scenario: Skipping a skippable step succeeds
- GIVEN `workflow.yaml` で step `research` が `skippable: true` と宣言されている
- WHEN ユーザーが `mspec skip research --reason "typo fix, no investigation needed"` を実行する
- THEN コマンドは終了コード 0 で完了する
- AND `research` ステップの state は以降 `skipped` として扱われる

### Requirement: FR-002 — `--reason` must be at least 10 characters

The system MUST require the `--reason <text>` flag for `mspec skip` and MUST fail with a non-zero exit code if the supplied reason is shorter than 10 characters.

#### Scenario: Missing reason fails
- GIVEN ユーザーが `mspec skip research` を `--reason` 無しで実行する
- WHEN コマンドが起動する
- THEN コマンドは非ゼロの終了コードで中断する

#### Scenario: Reason shorter than 10 characters fails
- GIVEN ユーザーが `mspec skip research --reason "short"` を実行する
- WHEN コマンドが reason をバリデートする
- THEN コマンドは非ゼロの終了コードで中断する
- AND スキップに伴う副作用 (プレースホルダ MD 生成・readme 追記・skip-log 記録) は発生しない

#### Scenario: Reason of 10 or more characters passes
- GIVEN ユーザーが `mspec skip research --reason "docs-only change"` を実行する
- WHEN コマンドが reason をバリデートする
- THEN バリデーションを通過し、後続のスキップ処理が継続される

### Requirement: FR-003 — Generate placeholder MD for each `produces` artifact

The system MUST, on successful `mspec skip`, generate a placeholder Markdown file for every entry declared in the skipped step's `produces` list, so that downstream steps relying on `requires` can still resolve their inputs.

#### Scenario: Placeholder is generated per produces entry
- GIVEN step `design` が `produces: [design.md, architecture-overview.md]` で `skippable: true`
- WHEN ユーザーが `mspec skip design --reason "trivial CSS tweak"` を実行する
- THEN `changes/<change-dir>/design.md` と `changes/<change-dir>/architecture-overview.md` の双方がプレースホルダ MD として生成される

#### Scenario: Placeholder carries the skipped-step header
- GIVEN `mspec skip` がプレースホルダ MD を生成する
- WHEN 生成されたファイルの先頭を読む
- THEN 1 行目に `<!-- mspec: skipped step -->` コメントが存在する
- AND 続いて `# Skipped: <step-id>`, `Reason: <reason>`, `Skipped at: <timestamp>`, `See: ../readme.md → ## Skipped Steps` が記載される

### Requirement: FR-004 — Append entry to `readme.md` `## Skipped Steps`

The system MUST append a single line in the format `- <step-id>: <reason> (skipped at <timestamp>)` to the `## Skipped Steps` section of the change's `readme.md` on every successful `mspec skip`.

#### Scenario: Skipped Steps section is updated
- GIVEN チェンジの `readme.md` に `## Skipped Steps` セクションが存在する
- WHEN ユーザーが `mspec skip quickstart --reason "docs only update"` を実行する
- THEN `readme.md` の `## Skipped Steps` 配下に `- quickstart: docs only update (skipped at <timestamp>)` の行が追記される

### Requirement: FR-005 — Record skip in `.mspec/cache/skip-log.json`

The system MUST persist every successful skip event to `.mspec/cache/skip-log.json` so that `mspec status` can surface `state: "skipped"` for the affected step.

#### Scenario: skip-log captures the event
- GIVEN `.mspec/cache/skip-log.json` が空または存在しない
- WHEN ユーザーが `mspec skip research --reason "trivial doc tweak"` を実行する
- THEN `.mspec/cache/skip-log.json` が作成または追記される
- AND エントリには step-id, reason, timestamp, change-dir が含まれる

#### Scenario: status reports `skipped` state
- GIVEN `mspec skip research --reason "trivial doc tweak"` が成功している
- WHEN ユーザーが `mspec status --change <name> --json` を実行する
- THEN `research` step の `state` フィールドは `"skipped"` である

### Requirement: FR-006 — Constitution Check is bypassed for skipped steps

The system MUST exempt placeholder MDs generated by `mspec skip` from the Constitution Check section requirement, and `mspec validate` MUST skip Constitution Check verification for any step whose state is `skipped`.

#### Scenario: Placeholder MD has no Constitution Check table
- GIVEN `mspec skip design --reason "docs-only change"` が成功している
- WHEN 生成された `design.md` プレースホルダを読む
- THEN ファイルには Constitution Check 表が含まれない

#### Scenario: validate does not fail on missing Check for skipped step
- GIVEN step `design` が `state: skipped` であり、対応するプレースホルダ MD が Constitution Check 表を持たない
- WHEN ユーザーが `mspec validate --change <name>` を実行する
- THEN コマンドは Constitution Check 欠落を理由に fail しない

### Requirement: FR-007 — `mspec questions --phase <step-id>` returns merged bank

The system MUST return the question bank for the given phase by merging the CLI-bundled default bank with the project-local override bank.

#### Scenario: Default bank is returned when no override exists
- GIVEN プロジェクトに `.mspec/questions/proposal.yaml` が存在しない
- WHEN ユーザーが `mspec questions --phase proposal --json` を実行する
- THEN CLI 同梱の `<cli-pkg>/templates/questions/proposal.yaml` の質問群がそのまま返される

#### Scenario: Project override merges with default
- GIVEN プロジェクトに `.mspec/questions/proposal.yaml` が存在し、デフォルトに無い ID の質問を 1 件含む
- WHEN ユーザーが `mspec questions --phase proposal --json` を実行する
- THEN デフォルト bank の質問とプロジェクト追加の質問の双方が結果に含まれる

#### Scenario: Same-ID override replaces default entry
- GIVEN プロジェクトの `.mspec/questions/proposal.yaml` がデフォルト bank と同一 ID の質問を再定義している
- WHEN ユーザーが `mspec questions --phase proposal --json` を実行する
- THEN 同一 ID の質問はプロジェクト側の定義で上書きされる

### Requirement: FR-008 — Bank covers the 9 Spec Kit categories

The system MUST recognise the following nine `category` values in bank YAML: `functional_scope`, `data_model`, `ux`, `nfr`, `integration`, `edge_cases`, `constraints`, `terminology`, `completion`.

#### Scenario: Unknown category is rejected
- GIVEN プロジェクトの `.mspec/questions/proposal.yaml` に `category: performance` という未知カテゴリを持つ質問がある
- WHEN ユーザーが `mspec questions --phase proposal --json` を実行する
- THEN コマンドは非ゼロの終了コードで中断する
- AND エラーメッセージは未知カテゴリ名を含む

#### Scenario: Returned questions retain category field
- GIVEN `mspec questions --phase proposal --json` がマージ済み bank を返す
- WHEN 出力 JSON を読む
- THEN 各質問エントリは 9 カテゴリのいずれかの値を持つ `category` フィールドを含む

### Requirement: FR-009 — Evaluate `when` condition for dynamic filtering

The system MUST evaluate each question's `when` field and exclude questions whose condition is unsatisfied from the returned bank.

#### Scenario: `when: always` is always included
- GIVEN bank に `when: always` の質問 `PRP-FS-001` が含まれる
- WHEN ユーザーが `mspec questions --phase proposal --json` を実行する
- THEN 結果に `PRP-FS-001` が含まれる

#### Scenario: Conditional question is hidden until trigger answered
- GIVEN bank に `when: "answers.PRP-FS-001 == '全く新規の機能'"` の質問 `PRP-FS-002` が含まれる
- AND 既存の回答ログに `PRP-FS-001` の回答が無い、または `全く新規の機能` 以外である
- WHEN ユーザーが `mspec questions --phase proposal --json` を実行する
- THEN 結果に `PRP-FS-002` は含まれない

#### Scenario: Conditional question is exposed when trigger matches
- GIVEN 回答ログに `PRP-FS-001 == "全く新規の機能"` が記録されている
- WHEN ユーザーが `mspec questions --phase proposal --json` を実行する
- THEN 結果に `PRP-FS-002` が含まれる

### Requirement: FR-010 — Support `options: dynamic` for runtime-generated choices

The system MUST recognise `options: dynamic` in bank YAML and resolve the choice list at runtime from the appropriate project context (e.g. existing capability names) before returning the bank.

#### Scenario: Dynamic options resolved from capability list
- GIVEN プロジェクトの `specs/` 配下に capability `theme-engine` と `search` のディレクトリが存在する
- AND bank の質問 `PRP-FS-002` が `options: dynamic` で定義されている
- WHEN ユーザーが `mspec questions --phase proposal --json` を実行する
- THEN 該当質問の `options` フィールドは `theme-engine`, `search` を含む配列に展開される

### Requirement: FR-012 — Bank lookup is 2-tier with package default and project override

The system MUST resolve question banks from exactly two locations, in order of precedence: project override `.mspec/questions/<step-id>.yaml` first, CLI-bundled default `<cli-pkg>/templates/questions/<step-id>.yaml` second, and MUST NOT copy the default bank into the project on `mspec init`.

#### Scenario: init does not materialise default bank
- GIVEN プロジェクトに `mspec init` を実行した直後である
- WHEN プロジェクトの `.mspec/` 配下を確認する
- THEN `.mspec/questions/` ディレクトリは作成されていない

#### Scenario: Missing phase bank returns empty set
- GIVEN CLI 同梱とプロジェクト側のいずれにも `templates/questions/<step-id>.yaml` が存在しない step を指定する
- WHEN ユーザーが `mspec questions --phase <step-id> --json` を実行する
- THEN コマンドは終了コード 0 で完了する
- AND 出力 JSON の `questions` 配列は空である
