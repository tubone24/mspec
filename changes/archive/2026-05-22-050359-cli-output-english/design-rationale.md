# Design Rationale: cli-output-english

## Context

`mspec upgrade` コマンドは npm registry からバージョンを取得してユーザーに表示するが、その出力メッセージが日本語でハードコードされている。`@mspec/cli` パッケージは国際的な開発者が利用するツールであり、CLI の出力が英語であることはツールの標準的な慣行に沿う。

調査の結果、`packages/cli/src/commands/upgrade.ts` の `upgradeCommand()` 関数に7箇所の日本語文字列が UTF-8 リテラルとして直接記述されており、i18n ライブラリは一切使われていないことが判明した。

## Decisions

### 直接置換を採用した理由

i18n ライブラリを導入せずソース内のリテラルを英語に置換する方式を採用した。理由は2つある。第一に、`@mspec/cli` は現時点で単一ロケール（英語）で動作すれば十分であり、多言語対応の要件は存在しない。第二に、ライブラリ追加はビルドパイプライン・バンドルサイズ・型定義に影響するため、今回の bugfix スコープを超える。

### テスト期待値の更新

文字列が変わるためユニットテスト・E2E テストの期待値も同時に更新する。テストを更新しないと CI が壊れ、変更の正しさを継続的に保証できなくなる。

## Alternatives Considered

- **i18next 等の i18n ライブラリを導入**: 将来の多言語対応に備えられるが、現時点では過剰。bugfix スコープを大きく超える。
- **dist/index.js を直接編集**: ビルド成果物を直接変更すると次回ビルドで上書きされるため却下。

## Trade-offs

- 採用案は将来的に別言語を追加する際、i18n ライブラリへの移行コストが発生する
- 一方で現在のシンプルな構造を維持でき、変更差分が最小に抑えられる

## Rejected Options

- **i18n ライブラリ導入**: bugfix モードの変更スコープに対して過剰。将来の要件が確定してから別チェンジで対応する。
- **dist/ 直接編集**: ビルド成果物を変更しても次回 build/publish で上書きされる。却下。
