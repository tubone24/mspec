## Source-of-Truth Regression Risk

### cli-delta-spec (SoT: `specs/cli-delta-spec/spec.md`)

- [ ] cli-delta-spec FR-007: FR-007 accepts `MUST / SHALL / SHOULD / MAY` as valid RFC 2119 keywords in generated stubs. The FR-011 change from `MUST` to `SHALL` satisfies FR-007's letter. Verify that no E2E or unit test snapshot asserts the exact string `The system MUST` as the stub output of `delta-init` — such a test would produce a false negative after the change. ⚠️
- [ ] cli-delta-spec FR-001–FR-010 (既存): All existing auto-numbering, section structure, H3/H4 heading, FR-ID collision, and placement requirements are unchanged. Confirm `delta-init.ts` edits are limited to the keyword substitution and no surrounding logic is inadvertently modified.

### cli-workflow-engine (SoT: `specs/cli-workflow-engine/spec.md`)

- [ ] cli-workflow-engine FR-003: FR-003 Scenario asserts `readme.md` is generated by `mspec new`. Adding `glossary.md` as a second produced file does not violate the Scenario text, but `continue-envelope.e2e.test.ts` marks the `new` step `done` solely by the presence of `readme.md`. If `workflow.yaml` or the status engine is updated to include `glossary.md` in `new`'s `produces` list, existing tests that write only `readme.md` will see the `new` step as not `done`, breaking downstream assertions. ⚠️
- [ ] cli-workflow-engine FR-005: FR-005 marks a step `done` only when every file in `produces` exists and passes `mspec validate`. Confirm `workflow.yaml` `new` step's `produces` field is not extended to include `glossary.md` without corresponding test updates, or that the design explicitly keeps `glossary.md` outside `produces` (side-effect, not a validated artifact).

### cli-init (SoT: `specs/cli-init/spec.md`)

- [ ] cli-init FR-007: FR-007 requires `mspec init --tools claude` to place `.claude/skills/mspec-*/SKILL.md` files. The FR-010 update modifies the *content* of three SKILL.md files. The placement requirement is not broken, but any snapshot test asserting the exact content of these SKILL.md files after `mspec init` would fail. Verify no such content snapshot exists or update it as part of implementation. ⚠️
- [ ] cli-init FR-002: FR-002 scaffolds `memory/constitution.md`. This change adds a new section to `constitution.md`. Confirm `mspec init --force` regenerates the template without the new RFC 2119 keyword semantics section (since the template and the live file are separate), and that `mspec init` tests do not assert exact constitution content.

### artifact-taxonomy SoT (`specs/artifact-taxonomy/spec.md`)

- [ ] artifact-taxonomy SoT は現在スタブ（`## Requirements` セクションが空）。`mspec archive` による FR-001〜FR-003 のマージが空の Requirements セクションへ正しく注入されることを確認する。archive CLI パーサーが既存 `### Requirement:` ヘッダを持たない spec を受け入れ FR-001 からシードできることを検証する。 ⚠️

### claude-integration (SoT: `specs/claude-integration/spec.md`)

- [ ] claude-integration FR-002: FR-002 requires each SKILL.md to have a `## Procedure` section. The FR-010 edits add content inside that section. Confirm the edits do not alter the `##` heading level or rename the section, which would cause FR-002's Scenario to fail.
- [ ] claude-integration FR-003: FR-003 requires the first step in every SKILL.md Procedure to be `mspec status --change <name> --json`. Confirm the EARS+Scenario instructions are inserted after step 1 (not prepended), so FR-003's ordering invariant is preserved.
- [ ] claude-integration FR-006: FR-006 requires SKILL.md to delegate structural validation to `mspec validate --change <name>` and not re-implement CLI logic. Confirm the EARS+Scenario additions are textual guidance only and do not embed inline Markdown validation logic.
