# Design Rationale: fix-anchor-change-dir-lookup

## Context

`mspec anchor-check` はプロジェクト内のすべてのソースファイルをスキャンして `@mspec-delta` アンカーブロックを発見し、参照先の change_dir やデルタスペックが存在するかを検証するコマンドである。

問題は `anchor.test.ts` にある。このファイルはアンカーパーサーのユニットテストであり、テストデータとして `@mspec-delta` を含む文字列をテンプレートリテラル内に書いている。`parseAnchors` は文字列リテラルをマスクしないため、テストデータが本物のアンカーとして認識され、存在しない `change_dir "2026-05-14-093015-apply-css"` への参照エラーとして報告される。

この問題は `anchor.test.ts` だけでなく、将来新しいテストが書かれるたびに再発しうる構造的な問題であるため、根本原因を修正することが重要である。

## Decisions

### テストファイル除外（Option A）を採用しなかった理由

当初の FR-018 は「テストファイルをスキャン対象から除外する」と記述されていたが、リサーチ段階でこの方針が SoT spec に違反することが判明した。`specs/cli-anchor/spec.md` の FR-004、FR-013、FR-014 はいずれも「テストファイルもスキャン対象に含む」と明記しており、プロジェクト内に13個以上の実アンカーを `//` 行コメントとして先頭に持つテストファイルが存在することも確認された。

Option A を採用すると、これら13個のテストファイルのアンカーが全て孤立（見えなくなる）し、`anchor-check` が誤って「正常」と報告する状態になる。これは SoT spec 違反かつ既存のワークフロー運用を壊す破壊的変更である。

### 文字列リテラルマスク（Option B）を採用した理由

Option B の根本的なアイデアは「文字列リテラルの内部にあるテキストは、そのファイルのソースコードの意味（コメントや実装）ではなくデータである」という洞察に基づく。`parseAnchors` は既に `blankOutFences`（コードフェンス）と `blankOutHtmlComments`（HTMLコメント）でこの原則を適用しており、テンプレートリテラルのマスクもこれと同じ原則の延長である。

変更の影響範囲が最小で、既存コードのスタイルと一致している点も採用の理由である。

### テンプレートリテラルのみをマスクする理由

シングルクォートとダブルクォートで囲まれた文字列は JavaScript/TypeScript において原則として複数行にまたがらない（バックスラッシュ行継続は例外的であり、mspec コードベースでは未使用）。複数行の `@mspec-delta` アンカーブロックを偽造できるのは事実上テンプレートリテラルのみである。したがって、バッククォートのみを対象にすれば問題が解決する。

## Alternatives Considered

- **全文字列リテラルをマスクする**: バッククォートだけでなく `'...'` と `"..."` も対象にする案。これらは複数行アンカーを偽造できないため不要であり、コメント内のシングルクォートを誤ってマスクするリスクがある。採用せず。
- **anchor-scanner でスキャン対象ファイルをフィルタする**: より上流（ファイル選択層）で制御する案。SoT spec との矛盾により不採用（Option A と同様）。
- **正規表現ベースのテンプレートリテラルマスク**: `blankOutHtmlComments` で使われているアプローチ。バッククォート内の `${}` ネストや `\`` エスケープを正規表現で正確に処理するのが非常に困難なため、状態マシンを採用。

## Trade-offs

- **`//` コメント内バッククォートによる誤マスク**: `// description with \`backtick\`` のような行コメントがあると、その後の複数行がマスクされる可能性がある。`SCAN_LINES_MAX = 30` の範囲内でこのパターンが実アンカーの直前に現れる確率は極めて低く、bugfix スコープとして許容する。
- **`${...}` ネストを追跡しない**: テンプレートリテラル内の `${expr}` を考慮しない単純実装。`expr` 内の文字列がネストしていても正しくマスクできない場合があるが、mspec の実使用ケースでは問題にならない。

## Rejected Options

- **Option A（テストファイル除外）**: SoT spec（FR-004/FR-013/FR-014）違反。13個の既存テストアンカーが孤立する。完全に却下。
- **`anchor.test.ts` のテストデータをコメント形式に書き直す**: テストファイルを変更するコストが高い。また、将来のテスト追加者が同じ問題を再発させる構造的な問題は解消されない。採用せず。
