# Design Rationale: web-ui-artifact-order-and-test-results

## Context

mspec Web UI のチェンジ詳細画面では、アーティファクト（proposal.md、design.md 等）が OS の `readdir` 順で表示されていた。これはファイルシステムの実装依存であり、ユーザーが現在のワークフロー進捗を把握しにくい状態を生んでいた。特に、変更ごとにアーティファクトが追加されるたびにソート順が変化するという問題があった。

テストリザルトについては、Playwright の内部 JSON フォーマット（`suites[].specs[].tests[].results[]`）を直接参照する設計になっており、テストフレームワークの内部構造に強く依存していた。また、チェンジをまたいで test-result-viewer が共有構造を参照する設計は、チェンジごとの独立した結果管理を困難にしていた。

チェックリスト項目とテスト結果の紐づきは、UI で一切提供されておらず、品質確認の文脈（どのテストがどの要件を検証しているか）が失われていた。

## Decisions

### アーティファクトソートをサーバー側で実施した理由

アーティファクトのソートはサーバー側（`routes/artifacts.ts`）で実施することにした。クライアント側（`ChangeDetail.tsx`）でソートする選択肢もあったが、API を利用するすべてのクライアント（将来の CLI ツール、CI インテグレーション等）に一貫したソート結果を提供できる点を重視した。クライアントはレスポンス配列をそのまま render するだけになり責務が明確になる。

ソートキーとして `workflow.yaml` を動的に読み込む案もあったが、リクエストごとのファイル IO コストを避けるため、定数配列として実装に埋め込む判断をした。ワークフローのステップ順変更はまれであり、その場合は定数配列の更新とともにコードレビューの対象とすることで対応する。

### `test-results.json` を新規ファイルとして設計した理由

既存の `.agent-runs.jsonl` にテストリザルトフィールドを追加する案があったが、agent-runner FR-003（Log Sanitization）が `.agent-runs.jsonl` の許可キーを 7 つに厳格に固定しており、テストリザルト追加は仕様違反になる。また、`.agent-runs.jsonl` は実行メタデータ専用の責務を持っており、関心の分離の観点からも不適切だった。

`test-results.json` を独立したファイルとして設計することで、テストリザルトデータの明確なオーナーシップ（CLI サブコマンドが書き込む）と、Web UI の参照先の一本化が実現できる。

### ステータス語彙をサーバー側アダプターで変換した理由

新仕様の `test-results.json` は `red / green / skip`、既存フロントエンド型は `pass / fail / skip` という語彙の不一致があった。フロントエンド全体を `green / red` にリネームする案もあったが、既存の E2E テスト（`data-testid="test-case-pass"` 等）への破壊的変更を避けるため、サーバー側変換層を選択した。これは Adapter パターンの適用であり、ストレージ層（JSON）とプレゼンテーション層（React コンポーネント）の語彙を独立させる。

### テスト名への FR 番号埋め込みを紐づけメカニズムとして選択した理由

テストとチェックリスト項目の紐づけには、（A）テスト名への FR 番号埋め込み命名規約、（B）手動記述、（C）サイドカー設定ファイルの3案があった。命名規約（A）を選択した理由は、テストファイルとチェックリスト項目の紐づきがテストコード自体から読み取れるため、別ファイルの管理コストが不要になるからである。`[FR-001] renders checklist` のような形式はテスト名の可読性も損なわない。

## Alternatives Considered

- **クライアント側ソート**: `ChangeDetail.tsx` でアーティファクトをソートする案。実装はシンプルだが、API の一貫性が失われる。将来の API 利用者（CI ツール等）が独自にソートを実装する必要が生じる
- **Playwright XML / JSON からの直接表示継続**: 既存の Playwright XML / JSON をそのまま参照し続ける案。Playwright の内部スキーマ変更に脆弱であり、チェックリスト紐づきの実現が困難
- **`green / red` への全面リネーム**: フロントエンド型も含めてすべてを `green / red` に統一する案。語彙の一貫性は高まるが、既存 E2E テストの大規模修正が必要になる
- **サイドカー設定ファイルによる紐づき管理**: 別途マッピングファイルを管理する案。柔軟性は高いが、テストファイルとの二重管理コストが発生する

## Trade-offs

- **定数配列によるソートキー**: `workflow.yaml` との二重化が発生する。ワークフロー変更時に定数配列の更新が必要。ただしその頻度は低く、コードレビューで対応可能と判断
- **サーバー側のチェックリスト読み込み**: `checklist.md` を API リクエストのたびに読み込む処理コストが発生。ローカル開発ツールのため許容範囲内
- **テスト名への FR 番号埋め込み**: 既存のテスト名を `[FR-NNN]` 形式に変更する一括リネームが必要。移行コストは一度限りで、以後はルールに従うだけで自動紐づきが機能する

## Rejected Options

- `.agent-runs.jsonl` へのテストリザルト追加: FR-003 違反のため却下
- フロントエンド側のスタックトレースマスク: ネットワーク経由で機密情報が露出するリスクがあるため、サーバー側でのマスクを優先
- Playwright カスタムレポータープラグインによる直接書き込み: Playwright への依存追加と、プラグイン管理コストが発生するため却下。CLI サブコマンドによる変換アプローチの方がシンプル
